# New Change Plan

Interactively scaffolds the `.changes/[slug]/` directory for a new change, working with the user to capture the *why* and *what* in `proposal.md` and (when warranted) the *how* in `design.md`.

The goal is a conversation, not a form dump. Ask focused questions, propose draft content, and let the user refine it. Do not leave the documents as empty templates — fill them in collaboratively before finishing.

## Steps

1. **Understand the change.** If the user already described what they want, start from that. Otherwise ask what change they want to plan. Keep probing until you can answer:
   - What problem is being solved, or what is the goal?
   - What is in scope, and explicitly what is out of scope?
   - Is this trivial (proposal only) or non-trivial (needs a design)?

   Ask only a few questions at a time. Use the previous answers to inform the next ones rather than front-loading a long questionnaire.

2. **Agree on the slug.** Use the argument passed to `/change-planning` if provided. Otherwise propose a short, lowercase, hyphen-separated slug based on the discussion (e.g. `revoke-sessions`, `user-data-store`) and confirm it with the user.

3. **Check for conflicts.** If `.changes/[slug]/` already exists, tell the user and stop.

4. **Draft `proposal.md` collaboratively.** Using what you have learned, write a real first draft — not blank headings. Fill in Problem & Goal, Scope, and Solution Outline with concrete content drawn from the conversation. Mark anything you are unsure about so the user can correct it, then ask for their feedback and revise.

5. **Decide on `design.md`.** Ask the user whether the change is non-trivial enough to need a design document.
   - If yes, draft `design.md` the same way — populate Overview and Architecture Changes from the discussion, and seed the Open Questions list with the genuine unknowns surfaced while planning.
   - If no, skip it. It can always be added later.

6. **Write the metadata.** Create `[slug].toml` with the agreed `name`, a real one-line `description`, and `status = "planning"`. Add an empty `[versions]` table if the repo uses versioned packages.

7. **Do not create `tasks.md`.** That is generated later with `/change-tasks` when implementation is imminent.

8. **Confirm and prompt.** Summarise what was created and remind the user:
   - Review and refine `proposal.md` and `design.md` — they are starting drafts.
   - Resolve the Open Questions in `design.md` before implementation.
   - Run `/change-tasks [slug]` when planning is complete and implementation is about to begin.

## Templates

Use these as the skeleton, but always populate them with content from the conversation rather than leaving them empty.

### `[slug].toml`

```toml
name = "[slug]"
description = "A brief one-line description of the change"
status = "planning"
```

If the repo uses versioned packages, add an empty `[versions]` table:

```toml
[versions]
```

### `proposal.md`

```markdown
# Proposal: [Title]

## Problem & Goal

## Scope

## Solution Outline
```

### `design.md`

```markdown
# Design: [Title]

## Overview

## Architecture Changes

## Questions

### Open

### Resolved

## Notes
```

## Document purposes

**`proposal.md`** — the *why* and *what*. Should stay concise and stable. Does not contain implementation detail.

**`design.md`** — the *how* at an architectural level. Bridges proposal to tasks. Needed for non-trivial changes; can be skipped for simple ones. If written, include at minimum: Overview, Architecture Changes, Questions, Notes.

**`tasks.md`** — the executable checklist. Generated by `/change-tasks`. Only created shortly before implementation begins to avoid stale decomposition.

**`[slug].toml`** — metadata. Status progresses: `planning` → `implementing` → `finished` → `merged`.
